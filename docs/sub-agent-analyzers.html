Security Analyzers

The specialized sub-agents that run on every pull request - XSS, SQLi, IDOR, SSRF, Mass Assignment, Secrets, and General Security.

How Analyzers Work

DryRun Security's multi-agent architecture deploys specialized sub-agents on every pull request. Each analyzer focuses on a specific class of vulnerability and uses Contextual Security Analysis to evaluate code in full context rather than matching patterns.

Analyzers run in parallel during a PR review. The Code Review Agent coordinates the sub-agents, collects their findings, validates exploitability, and presents a consolidated report in the PR comment. Each analyzer can also call just-in-time research sub-agents to look up framework-specific behavior, CVE details, or license information as needed.

SQL Injection (SQLi) Analyzer

Detects unsafe query composition across languages by tracing data flow from user input sources to query execution sinks. The analyzer distinguishes between genuinely unsafe dynamic queries and parameterized queries, ORM abstractions, and other safe patterns that happen to involve user-derived values.

Covers SQL injection in raw queries, stored procedures, ORM misuse, and query builder patterns across all supported languages.

Cross-Site Scripting (XSS) Analyzer

Finds unescaped user input rendered in HTML output. Goes beyond simple innerHTML detection to cover framework-specific pitfalls - template engine auto-escaping bypass, React dangerouslySetInnerHTML, Vue v-html directives, and server-side rendering contexts where escaping rules differ.

The analyzer evaluates whether the rendering context (HTML body, attribute, script, URL) matches the escaping applied, catching cases where input is escaped for one context but used in another.

Insecure Direct Object Reference (IDOR) Analyzer

Surfaces broken object-level authorization by tracing data paths from request parameters to data access. The analyzer looks for cases where a user-supplied identifier (ID, slug, filename) is used to retrieve or modify a resource without verifying that the requesting user has access to that resource.

This goes well beyond checking whether an endpoint exists - the analyzer reasons about authorization middleware, access control decorators, and ownership checks in the data access layer.

Mass Assignment Analyzer

Flags unsafe binding patterns that allow user input to overwrite sensitive model fields or escalate privileges. Common in web frameworks that automatically map request parameters to model attributes (Rails params.permit, Django model forms, Express body parsing).

The analyzer identifies when user-controlled input reaches a model update without an explicit allowlist of permitted fields, and evaluates whether any of the unprotected fields are security-sensitive (roles, permissions, account status, pricing).

Server-Side Request Forgery (SSRF) Analyzer

Inspects outbound HTTP calls, DNS lookups, and other network operations for user-controlled target URLs or hostnames. The analyzer checks for missing URL validation, allowlist enforcement, and internal network access restrictions.

Covers direct SSRF (user input in URL), blind SSRF (request made but response not returned to user), and partial SSRF (user controls part of the URL path or query parameters).

Secrets Analyzer

Stops committed API keys, tokens, database passwords, private keys, and other credentials at PR time. Unlike simple regex scanners, the Secrets Analyzer evaluates context - distinguishing between real credentials and test fixtures, example configurations, public documentation keys, and placeholder values.

When a confirmed secret is found, the finding includes the file path and line number with a plain-language explanation of the risk, without reproducing the credential value in the report. See Secrets Detection for the full details.

General Security Analyzer (GSA)

The GSA is the broadest analyzer, covering security issues that do not fall neatly into a single vulnerability class:

  • Authentication and authorization gaps - missing auth middleware, inconsistent access controls, broken session management
  • Risky cryptography - weak algorithms, hardcoded IVs, improper key management
  • Debug artifacts - debug endpoints, verbose error messages, development-mode settings left in production code
  • Unsafe deserialization - untrusted data passed to deserialization functions without validation
  • Leaky error handling - stack traces, internal paths, or database details exposed in error responses
  • Missing rate limiting - authentication endpoints, API routes, and form submissions without throttling
  • Security misconfigurations - permissive CORS, missing security headers, unsafe defaults
  • General logic flaws - race conditions, time-of-check/time-of-use issues, and other semantic vulnerabilities

Core Code Policies

In addition to the specialized analyzers, DryRun runs a set of Core Code Policies on every PR. These cover the OWASP Top 10 and other well-established vulnerability patterns using the same contextual analysis approach. Core Code Policies are always active and require no configuration.

For organization-specific rules beyond the Core Code Policies, see Natural Language Code Policies.

Configuring Analyzers

Each Security Analyzer can be individually configured per repository through the Configurations page. In the Code Security Agents section of a configuration, every analyzer has three controls:

  • Blocking - When enabled, a finding from this analyzer creates a GitHub status check that must pass before the PR can be merged. The check appears under the analyzer's name (e.g., "Secrets Analyzer").
  • Silent Mode - When enabled, the analyzer still runs and records findings in the Risk Register, but does not post them in the PR comment. Useful for evaluating an analyzer before turning it on for developers.
  • Risk Level - Sets the default severity label for findings from this analyzer. Options are Risky, Fail, or Info. This controls how findings appear in the PR comment and Risk Register.

These settings let teams gradually roll out analyzers - for example, starting in Silent Mode to evaluate signal quality before enabling Blocking for enforcement.
