PR Variant Analysis

Behavioral analysis and multi-agent PR review architecture.

Git Behavioral Analysis

DryRun Security constructs a Git Behavioral Graph before its AI agent reads a single line of code - analyzing commit history across five behavioral axes to steer the scanner toward the code that matters most.

The techniques described here are grounded in Adam Tornhill's Your Code as a Crime Scene (2nd ed., Pragmatic Programmers, 2024). DryRun Security engineered these forensic principles into a pipeline that steers an AI agent with deterministic precision. Read the full blog post for additional context.

Why Git History Matters for Security

Traditional static analysis (SAST) using abstract syntax tree (AST) parsing and call graphs is well studied, but these scanners lack a fundamental dimension of context: the human element. Vulnerabilities are rarely just syntactical errors - they are the byproduct of diffuse ownership, shifting requirements, and knowledge decay.

LLM context windows are finite, and agentic loops are computationally expensive. You cannot feed a giant monorepo into an LLM and ask it to find bugs. While DryRun Security already uses a hybrid LLM-driven context gathering and SAST-style multi-agent architecture to surface traditional vulnerabilities, hunting non-traditional structural flaws requires an additional layer. For deeper, complex logic issues, the agent needs a deterministic, high-signal heuristic to prioritize its attention.

The Git Behavioral Graph provides exactly this. Before our agent reads any code, it analyzes the repository's commit history across five distinct behavioral axes.

The Data Ingestion Engine

Parsing git history across a decade-old monorepo is notoriously memory-intensive. Loading the entire commit tree into memory guarantees out-of-memory (OOM) crashes.

DryRun Security's harvester uses an asynchronous subprocess streaming architecture. It pipes git log --numstat with custom delimiters into a generator function, yielding, parsing, and evaluating a single commit block at a time - resolving complex git brace rename notations on the fly while maintaining a strictly bounded memory footprint.

The Five Behavioral Axes

1. Code Churn: Quantifying Diffuse Ownership

Code churn is mathematically one of the strongest historical predictors of vulnerability density. The pipeline extracts total revisions, distinct contributor counts, and cumulative line additions and deletions. Two heuristic anomalies are particularly accurate:

  • The "Many-Hands" Anti-Pattern: When the number of distinct authors scales linearly with the commit count, ownership becomes completely diffuse. No single engineer holds a comprehensive mental model of the module's security boundaries.
  • Security Contract Renegotiation: When a file exhibits high churn velocity where cumulative deleted lines approximate cumulative added lines, it indicates continuous heavy rewriting - a classic signature of a module whose design is fundamentally unstable and lacks a clean architectural reset.

AI coding agents (Codex, Claude, etc.) do not improve the situation here. They are not humans with long-term knowledge and storage. They have a point-in-time concept of the changing code with discrete directions and narrow focus, and the range of contributors can widen dramatically with increased shipping velocity.

2. Temporal Coupling: Mapping Invisible Dependencies

Attackers exploit the invisible gaps between microservices. To find these, DryRun Security uses temporal coupling - a forensic graph technique that identifies files that frequently change together in the same changeset, regardless of whether a formal AST import relationship exists.

This is modeled as a bipartite graph of commits to files. The coupling degree is calculated as the ratio of shared revisions to the average individual revision count of the pair.

To keep the scanner performant, strict pruning heuristics are applied:

  • Changeset Caps: Massive repo-wide refactors (e.g., license header sweeps) are filtered out by ignoring any commit touching more than 30 files.
  • Cold File Pruning: A file pair must have at least 5 individual revisions. This prevents a cold file from generating a spurious 100% coupling degree after a single co-commit.
  • Aggressive Garbage Collection: Every 5,000 commits, the engine pauses to prune pairs with only a single co-commit - because of the minimum revision threshold, these can mathematically never reach the 50% target threshold.

3. Knowledge Decay: Age and Ownership Concentration

Age and ownership determine whether anyone actively understands a critical module. Two metrics are combined:

  • Age in Months: Calculated by diffing the file's last modified timestamp against current UTC time. High-age files often rely on deprecated cryptographic wrappers or broken standards (MD5, HTTP Basic Auth) that were patched everywhere else but forgotten here.
  • Ownership Concentration: The fraction of a file's total surviving code attributed to its primary contributor. If a single developer owns over 90% of a heavily modified file, it acts as a knowledge silo - if they leave, the implicit security context evaporates.

4. Temporal Anomalies: Circular Statistics for Threat Hunting

To identify compromised CI pipelines or insider threats, DryRun Security flags commits made at statistically unusual hours. Standard linear math fails on clocks - the linear mean of 23:00 and 01:00 is incorrectly calculated as 12:00, destroying the signal.

Instead, commit timestamps are mapped onto a unit circle using circular statistics. The decimal hour is converted to an angle in radians, and the mean of the sine and cosine components across an author's history is computed to find their true circular mean, along with the circular standard deviation based on shortest angular distances.

If a commit's angular distance from the author's circular mean exceeds 3.0 standard deviations, it triggers an anomaly. To filter noise, a baseline commit history per author is required, and anomalies during standard local business hours (9 AM - 5 PM) are ignored.

5. Intent Mining: The Developer Confessional

Developers narrate their architectural shortcuts in commit messages. Keywords like "bypass," "hardcode," or "remove before" function as explicit confessions of technical debt.

Making this actionable for an AI agent is a data projection problem. Historically interesting hacks on files that no longer exist waste the LLM's finite context window. The pipeline strictly intersects regex matches with the output of git ls-files. If a commit confesses to a "quick fix" and the files modified in that specific commit still exist in the current working tree, the agent is directed to analyze those live paths for latent bypass surfaces.

To prevent sensitive data leaks into the LLM context, all matched commit messages are routed through a dedicated secrets sanitizer before being passed to the planner.

How the Agent is Orchestrated

Classic code quality metrics like cyclomatic complexity, code churn, and temporal coupling are traditionally used to predict maintenance burden. From an AppSec perspective, maintenance burden is computationally indistinguishable from security risk. Where code quality degrades, structural entropy rises, and vulnerabilities breed.

A seasoned human security auditor does not read a monorepo linearly from main.py. They check git blame on the cryptographic wrappers, look for the auth module everyone hot-patches but nobody owns, and zero in on the files littered with "TODO: fix bypass later" comments. The Git Behavioral Graph explicitly digitizes and scales that human intuition into a deterministic data structure.

Large Language Models are inherently stochastic engines. If deployed naively against a massive codebase, they suffer from context fragmentation, attention decay, and hallucination. By pre-processing the repository into a behavioral artifact, DryRun Security fundamentally alters the agent's execution model: deterministic, mathematically rigorous algorithms tightly bound the search space, and only then is the LLM's stochastic reasoning unleashed to evaluate the semantics within that bounded space.

The agent is no longer asked to blindly "find vulnerabilities" in a vacuum. It receives a prioritized, JSON-structured map of human risk - pointing its finite context window precisely at the highly-coupled, heavily-churned, late-night architectural hacks. This synthesis of deterministic graph analysis and agentic semantic reasoning is what allows DryRun Security to scale elite, human-level security auditing across enterprise codebases.

Privacy and Developer Trust

Behavioral analysis is designed to augment security findings, not to monitor individual developers. The analysis is used to prioritize and contextualize security findings - never to generate reports about individual developer behavior or to create surveillance capabilities. The goal is better security outcomes, not productivity monitoring.

Security Analyzers

Investigate from Insights

Variant analysis can be triggered directly from the Insights page using the Investigate PR action. This allows you to seamlessly move from a high-level security question to a deep investigation of a specific pull request, with the full context of the variant analysis applied.

How Analyzers Work

DryRun Security's multi-agent architecture deploys specialized sub-agents on every pull request. Each analyzer focuses on a specific class of vulnerability and uses Contextual Security Analysis to evaluate code in full context rather than matching patterns.

Analyzers run in parallel during a PR review. The Code Review Agent coordinates the sub-agents, collects their findings, validates exploitability, and presents a consolidated report in the PR comment. Each analyzer can also call just-in-time research sub-agents to look up framework-specific behavior, CVE details, or license information as needed.

SQL Injection (SQLi) Analyzer

Detects unsafe query composition across languages by tracing data flow from user input sources to query execution sinks. The analyzer distinguishes between genuinely unsafe dynamic queries and parameterized queries, ORM abstractions, and other safe patterns that happen to involve user-derived values.

Covers SQL injection in raw queries, stored procedures, ORM misuse, and query builder patterns across all supported languages.

Cross-Site Scripting (XSS) Analyzer

Finds unescaped user input rendered in HTML output. Goes beyond simple innerHTML detection to cover framework-specific pitfalls - template engine auto-escaping bypass, React dangerouslySetInnerHTML, Vue v-html directives, and server-side rendering contexts where escaping rules differ.

The analyzer evaluates whether the rendering context (HTML body, attribute, script, URL) matches the escaping applied, catching cases where input is escaped for one context but used in another.

Insecure Direct Object Reference (IDOR) Analyzer

Surfaces broken object-level authorization by tracing data paths from request parameters to data access. The analyzer looks for cases where a user-supplied identifier (ID, slug, filename) is used to retrieve or modify a resource without verifying that the requesting user has access to that resource.

This goes well beyond checking whether an endpoint exists - the analyzer reasons about authorization middleware, access control decorators, and ownership checks in the data access layer.

Mass Assignment Analyzer

Flags unsafe binding patterns that allow user input to overwrite sensitive model fields or escalate privileges. Common in web frameworks that automatically map request parameters to model attributes (Rails params.permit, Django model forms, Express body parsing).

The analyzer identifies when user-controlled input reaches a model update without an explicit allowlist of permitted fields, and evaluates whether any of the unprotected fields are security-sensitive (roles, permissions, account status, pricing).

Server-Side Request Forgery (SSRF) Analyzer

Inspects outbound HTTP calls, DNS lookups, and other network operations for user-controlled target URLs or hostnames. The analyzer checks for missing URL validation, allowlist enforcement, and internal network access restrictions.

Covers direct SSRF (user input in URL), blind SSRF (request made but response not returned to user), and partial SSRF (user controls part of the URL path or query parameters).

Secrets Analyzer

Stops committed API keys, tokens, database passwords, private keys, and other credentials at PR time. Unlike simple regex scanners, the Secrets Analyzer evaluates context - distinguishing between real credentials and test fixtures, example configurations, public documentation keys, and placeholder values.

When a confirmed secret is found, the finding includes the file path and line number with a plain-language explanation of the risk, without reproducing the credential value in the report. See Secrets Detection for the full details.

General Security Analyzer (GSA)

The GSA is the broadest analyzer, covering security issues that do not fall neatly into a single vulnerability class:

  • Authentication and authorization gaps - missing auth middleware, inconsistent access controls, broken session management
  • Risky cryptography - weak algorithms, hardcoded IVs, improper key management
  • Debug artifacts - debug endpoints, verbose error messages, development-mode settings left in production code
  • Unsafe deserialization - untrusted data passed to deserialization functions without validation
  • Leaky error handling - stack traces, internal paths, or database details exposed in error responses
  • Missing rate limiting - authentication endpoints, API routes, and form submissions without throttling
  • Security misconfigurations - permissive CORS, missing security headers, unsafe defaults
  • General logic flaws - race conditions, time-of-check/time-of-use issues, and other semantic vulnerabilities

Core Code Policies

In addition to the specialized analyzers, DryRun runs a set of Core Code Policies on every PR. These cover the OWASP Top 10 and other well-established vulnerability patterns using the same contextual analysis approach. Core Code Policies are always active and require no configuration.

For organization-specific rules beyond the Core Code Policies, see Custom Code Policies.

Configuring Analyzers

Each Security Analyzer can be individually configured per repository through the Configurations page. In the Code Security Agents section of a configuration, every analyzer has three controls:

  • Blocking - When enabled, a finding from this analyzer creates a GitHub status check that must pass before the PR can be merged. The check appears under the analyzer's name (e.g., "Secrets Analyzer").
  • Silent Mode - When enabled, the analyzer still runs and records findings in the Risk Register, but does not post them in the PR comment. Useful for evaluating an analyzer before turning it on for developers.
  • Risk Level - Sets the default severity label for findings from this analyzer. Options are Risky, Fail, or Info. This controls how findings appear in the PR comment and Risk Register.

These settings let teams gradually roll out analyzers - for example, starting in Silent Mode to evaluate signal quality before enabling Blocking for enforcement.

Multi-Agent Architecture

Why Multi-Agent

DryRun Security uses a multi-agent architecture rather than a single monolithic model. Each agent specializes in a specific aspect of security analysis, and agents collaborate to produce comprehensive findings. This design mirrors how a senior security team works - one person focuses on authentication, another on injection, another on infrastructure - and the team's combined expertise exceeds what any individual could achieve.

The alternative - running all security analysis through a single model or pipeline - limits depth. A monolithic system must balance breadth against the context window and reasoning capacity available for any single vulnerability class. Multi-agent design removes this constraint.

Primary Agents

Code Review Agent

The Code Review Agent is the primary agent that runs on every pull request. It coordinates the overall review process: retrieving the PR diff and surrounding context, dispatching specialized sub-agents, collecting and validating findings, and posting the consolidated result as a PR comment and check.

The Code Review Agent also runs DryRun's Core Code Policies - a set of built-in checks covering the OWASP Top 10 and common vulnerability patterns. These are always active and require no configuration.

Custom Policy Agent

The Custom Policy Agent enforces your organization's Custom Code Policies alongside the standard SAST checks. When you write a policy like "no new API endpoints without authorization enforcement," the Custom Policy Agent interprets that policy and evaluates the PR against it, regardless of language or framework.

The Custom Policy Agent can call specialized sub-agents for SCA lookups, codebase analysis, and just-in-time research to fully evaluate whether a policy is satisfied.

DeepScan Agent

The DeepScan Agent performs full-repository security analysis on demand. While the Code Review Agent operates on PR diffs, the DeepScan Agent analyzes the entire repository tree at a given commit. This enables it to find vulnerabilities that span many files, were introduced gradually across multiple commits, or existed before DryRun Security was installed.

Codebase Insight Agent

The Codebase Insight Agent powers Code Security Intelligence - natural language queries about your codebase's security posture. Instead of navigating dashboards and exports, you ask questions like "what new API endpoints were added this week?" or "show me a chart of risky alerts by repo" and get precise, contextual answers.

Specialized Sub-Agents

Primary agents dispatch specialized sub-agents to handle specific vulnerability classes. Each sub-agent focuses on one area and uses Contextual Security Analysis for high-signal, low-noise findings:

  • SQLi Analyzer - SQL injection across languages, tracing sources to sinks
  • XSS Analyzer - cross-site scripting including framework-specific templating pitfalls
  • IDOR Analyzer - broken object-level authorization via data path tracing
  • Mass Assignment Analyzer - unsafe model binding from user input
  • SSRF Analyzer - server-side request forgery via user-controlled URLs
  • Secrets Analyzer - credentials in code, distinguished from test fixtures
  • General Security Analyzer - auth gaps, risky crypto, debug artifacts, deserialization, and more

See Security Analyzers for detailed documentation on each sub-agent.

How Agents Coordinate

When a PR is opened or updated, the flow works as follows:

  1. The Code Review Agent receives the PR event and retrieves the diff plus relevant surrounding files for context
  2. The agent dispatches specialized sub-agents in parallel, each analyzing the changes for their specific vulnerability class
  3. Sub-agents may call just-in-time research to look up framework behavior, CVE details, or license information
  4. The Custom Policy Agent evaluates any configured Custom Code Policies against the changes
  5. Findings from all agents are collected, validated for exploitability, and deduplicated
  6. The consolidated result is posted as a PR comment with findings, code references, and remediation guidance

Model-Independent Verification

Agents cross-check each other's findings to reduce the risk of any single model hallucinating or producing inconsistent results. This multi-agent verification is a key reason DryRun Security achieves high consistency and reliability in production environments. See Model-Independent Verification for more detail.
