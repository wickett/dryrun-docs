# AGENTS.md - DryRun Security Documentation

## Project Overview

Static documentation site for DryRun Security. Built with vanilla HTML/CSS/JS and a Python static site generator (`build.py`). Deployed to GitHub Pages at `docs.dryrun.security`.

## Build & Test

```bash
# Regenerate all documentation pages
python3 build.py

# Run unit tests
python3 -m pytest tests/ -v

# Run Playwright visual tests (requires: npx playwright install chromium)
npx playwright test
```

## Architecture

- `build.py` - Single-file static site generator; defines all page content as Python data structures and generates HTML
- `style.css` - All styles matching the DryRun Security brand (dark theme, `#C8FF09` accent, Inter + JetBrains Mono fonts)
- `app.js` - Mobile sidebar toggle, TOC scroll tracking, smooth scrolling, index search, code block copy buttons
- `index.html` - Landing page with hero, search, quick-start links, and section cards (generated by `build.py`)
- `docs/*.html` - Individual documentation pages (generated by `build.py`)
- `assets/` - Logo SVG and favicon

### Page Generation Flow

1. `build.py` defines `SECTIONS` (ordered dict of section name ŌåÆ page list) and `PAGES` (dict of slug ŌåÆ page data)
2. Each page has: `title`, `description`, `section`, `slug`, and `content` (raw HTML)
3. The generator creates a shared sidebar from `SECTIONS`, highlights the active page, extracts `h2`/`h3` for the TOC, and wires prev/next links
4. Output: `index.html` at root, `docs/{slug}.html` for each page, `sitemap.xml`, `robots.txt`

## Coding Conventions

- All HTML interpolation in `build.py` MUST use the `esc()` helper (wraps `html.escape`) for string values. Never interpolate raw strings into HTML.
- Internal links between pages use relative paths (`./`, `../`). No hardcoded domains.
- External links to dryrun.security, G2, LinkedIn etc. use full URLs with `target="_blank" rel="noopener noreferrer"`.
- Brand name is "DryRun Security" (no space between Dry and Run). "DryRun" shorthand only after full name appears twice.
- CSS custom properties match the capability matrix site exactly (`--bg-primary`, `--accent`, `--green`, etc.).

## Writing Style

- Never use em dashes (ŌĆö) in any content. Replace with a colon, comma, or reword the sentence. This applies to all pages, descriptions, table cells, headings, and any other content.

## Security

This repo is monitored by [DryRun Security](https://dryrun.security). Every PR is reviewed by DryRun Security's analyzers including:

- Cross-Site Scripting Analyzer
- SQL Injection Analyzer
- IDOR Analyzer
- Secrets Analyzer
- Server-Side Request Forgery Analyzer
- General Security Analyzer
- Code Policies

### Security Remediation

When DryRun Security flags a finding on a PR, use the [DryRun Security Remediation skill from Tessl](https://tessl.io/registry/dryrunsecurity/remediation) to understand and fix the issue:

```bash
tessl install dryrunsecurity/remediation
```

The skill enables AI coding agents to:

1. Parse DryRun Security findings from PR comments
2. Research authoritative sources (framework docs, OWASP standards)
3. Apply minimal, contextual fixes that match existing codebase patterns
4. Verify the fix addresses the finding without introducing regressions

**Remediation workflow:**
1. Read the DryRun Security comment on the PR to understand the finding
2. Identify the affected code and the vulnerability class (XSS, injection, etc.)
3. Research the correct fix using framework-specific documentation
4. Apply the fix - prefer `esc()` for output encoding in `build.py`
5. Re-run `python3 build.py` to regenerate all pages
6. Push and verify DryRun Security checks pass (no new comments, all checks green)

### Verifying DryRun Security Results

Before merging any PR:
1. All DryRun Security check runs must show `SUCCESS`
2. Any previous DryRun Security issue comments must be resolved (deleted/updated by the bot)
3. No new DryRun Security comments on the latest push

## Testing

### Unit Tests (`tests/test_build.py`)
- Data integrity: all pages have required fields (title, description, section, slug, content)
- HTML escaping: verify `esc()` is applied to all interpolated values
- Generated file validation: correct structure, DOCTYPE, favicon, header, footer
- Relative path verification: no hardcoded domains in internal links
- Sidebar consistency: every page appears in the sidebar
- TOC generation: verify heading extraction works correctly
- Prev/next navigation: verify link chain is complete and correct

### Visual Tests (`tests/visual.spec.js`)
- Desktop (1440x900) and mobile (390x844) viewport screenshots
- Verify header, sidebar, footer render correctly
- Verify sidebar active state highlights current page
- Verify TOC is visible on desktop, hidden on mobile
- Verify mobile sidebar toggle works
- Verify landing page cards and search render
- Verify prev/next navigation exists on doc pages

## Content Updates

All page content lives in `build.py` as Python data structures. To add or edit a page:

1. Add/edit the page dict in the `PAGES` definition (with `title`, `description`, `section`, `slug`, `content`)
2. Add the slug to the appropriate section in the `SECTIONS` ordered dict
3. Run `python3 build.py` to regenerate all pages
4. Verify with `python3 -m pytest tests/ -v`
